# Database Synchronization Guide

Complete guide for syncing data between local and production Supabase instances.

## Overview

The Design Systems MCP uses a **local-first development workflow** where you:
1. Develop and test locally with Supabase CLI
2. Ingest new content into your local database
3. Validate data integrity locally
4. Sync to production when ready

This approach ensures:
- ✅ Data consistency between environments
- ✅ Safe testing before production deployment
- ✅ Ability to work offline
- ✅ Fast local development
- ✅ Production database protection

## Prerequisites

### 1. Local Supabase Setup
```bash
# Install Supabase CLI
brew install supabase/tap/supabase

# Initialize and start local instance
supabase init
supabase start
```

### 2. Environment Configuration

Add production credentials to your `.env` file:

```bash
# Local Supabase (auto-generated by supabase start)
SUPABASE_URL=http://127.0.0.1:54321
SUPABASE_ANON_KEY=your-local-anon-key
SUPABASE_SERVICE_KEY=your-local-service-key

# Production Supabase (from cloud dashboard)
SUPABASE_PROD_URL=https://your-project-ref.supabase.co
SUPABASE_PROD_SERVICE_KEY=your-production-service-role-key

# OpenAI (required for embeddings)
OPENAI_API_KEY=sk-proj-your-key
```

**Get Production Credentials:**
1. Go to [Supabase Dashboard](https://app.supabase.com)
2. Select your project
3. Settings → API
4. Copy `URL` and `service_role` key

### 2. Pull from Production (Download)

Download production data to your local instance:

```bash
# Dry-run mode (preview changes)
npm run sync:pull -- --dry-run

# Execute sync (incremental by default)
npm run sync:pull

# Full sync (overwrites all local data)
npm run sync:pull -- --full

# Verbose output
npm run sync:pull -- --verbose
```

**Features:**
- ✅ Incremental sync (only downloads new entries)
- ✅ Preserves local changes
- ✅ Validates data integrity
- ✅ Progress reporting

**When to Use:**
- Initial setup: Download production data for local development
- Staying current: Pull new production content before adding more
- Testing: Sync production data to test against real content

### 3. Push to Production (Upload)

Upload local data to production:

```bash
# Dry-run mode (ALWAYS START HERE)
npm run sync:push -- --dry-run

# Execute sync (requires --confirm for safety)
npm run sync:push -- --confirm

# Full sync (overwrites all production data)
npm run sync:push -- --confirm --full

# Verbose output
npm run sync:push -- --confirm --verbose

# Skip validation (not recommended)
npm run sync:push -- --confirm --no-validate
```

**Safety Features:**
- ✅ Requires `--confirm` flag to prevent accidents
- ✅ Validates all entries before upload
- ✅ Incremental sync (only uploads new entries)
- ✅ Checks embedding dimensions (1536)
- ✅ Verifies required fields

**When to Use:**
- New content: After ingesting and validating new design systems content
- Updates: After fixing or improving existing content
- Deployment: When ready to make content available to users

## Sync Modes

### Incremental Sync (Default)

**Behavior:** Only syncs entries that don't exist in the target database.

**Advantages:**
- ✅ Fast (only transfers new data)
- ✅ Safe (preserves existing data)
- ✅ Efficient (minimal bandwidth)

**Use When:**
- Regular updates
- Adding new content
- Syncing frequently

**Example:**
```bash
# Pull only new production entries
npm run sync:pull

# Push only new local entries
npm run sync:push -- --confirm
```

### Full Sync (Force)

**Behavior:** Syncs all entries, overwriting existing data.

**Advantages:**
- ✅ Ensures complete consistency
- ✅ Fixes data corruption
- ✅ Resets to known state

**Use When:**
- Initial setup
- Fixing data issues
- Major migrations

**Example:**
```bash
# Overwrite all local data with production
npm run sync:pull -- --full

# Overwrite all production data with local
npm run sync:push -- --confirm --full
```

## Troubleshooting

### Issue: "Production credentials not found"

**Solution:** Add production credentials to `.env`:

```bash
SUPABASE_PROD_URL=https://your-project.supabase.co
SUPABASE_PROD_SERVICE_KEY=your-service-key
```

### Issue: Sync fails with "Invalid embedding dimension"

**Cause:** Embeddings are not 1536 dimensions (text-embedding-3-small).

**Solution:**
1. Check your OpenAI model in `.env`
2. Re-ingest content with correct embedding model:
```bash
npm run ingest:vectors
```

### Issue: "Databases have diverged"

**Cause:** Both databases have unique content that conflicts.

**Solutions:**

**Option A:** Keep production as source of truth:
```bash
# Backup local first
npm run sync:compare > local-backup.txt

# Pull production data
npm run sync:pull -- --full
```

**Option B:** Merge both databases:
```bash
# Pull production to local (incremental)
npm run sync:pull

# Then push local changes to production
npm run sync:push -- --confirm
```

**Option C:** Manual reconciliation:
```bash
# Compare and review differences
npm run sync:compare

# Manually decide which entries to keep
# Delete unwanted entries from local or production
# Then sync incrementally
```

### Issue: Slow sync performance

**Causes:**
- Large number of entries
- Large embedding vectors
- Network latency

**Solutions:**
1. Use incremental sync (default)
2. Compress embeddings in transit (future feature)
3. Sync during off-peak hours

### Issue: Sync succeeds but validation fails

**Possible Causes:**
- Chunks missing
- Embeddings corrupted
- Metadata incomplete

**Solution:**
```bash
# Run validation to identify specific issues
npm run validate:ingestion

# Re-ingest problematic entries
npm run ingest:vectors
```

## Security Considerations

- ⚠️ Never commit `.env` to version control
- ⚠️ Use different keys for local/production

- ⚠️ Only authorized team members should have `SUPABASE_PROD_SERVICE_KEY`

### Automated Sync

Set up automated syncs with cron or CI/CD:

```bash
# .github/workflows/sync-db.yml
name: Sync Databases
on:
  schedule:
    - cron: '0 2 * * *'  # Daily at 2 AM

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm install
      - run: npm run sync:pull
      - run: npm run validate:ingestion
```

## FAQ

**Q: Can I sync in both directions at once?**
A: No, sync one direction at a time to avoid conflicts.

**Q: What happens if sync fails midway?**
A: Incremental sync is safe - re-run to continue from where it stopped.

**Q: Can I sync specific entries only?**
A: Not yet. Use incremental sync or manual filtering.

**Q: How do I undo a sync?**
A: Pull from production (if it's your source of truth) or restore from backup.

**Q: Is sync atomic?**
A: No. Each entry is committed individually. Use dry-run to preview changes.

**Q: Can multiple people sync simultaneously?**
A: Yes, but coordinate to avoid conflicts. Last write wins.
